# EzeeChatBot — Minimal RAG API

This repository contains the backend implementation for **EzeeChatBot**, a minimal Retrieval-Augmented Generation (RAG) chatbot API. It allows users to upload content (PDFs, URLs, or plain text) to instantly create an isolated knowledge base, and then chat with it using an LLM.

## Setup Instructions

1. **Clone the repository and enter the directory**:
   ```bash
   git clone <your-repo-url>
   cd EZEE_CHATBOT
   ```

2. **Set up a Virtual Environment**:
   ```bash
   python -m venv .venv
   source .venv/bin/activate
   ```

3. **Install Dependencies**:
   ```bash
   pip install -r requirements.txt
   ```

4. **Environment Variables**:
   Ensure you have a `.env` file in the root directory with the necessary API keys.
   ```
   GROQ_API_KEY="your_groq_key_here"
   HUGGINGFACE_API_KEY="your_huggingface_key_here"
   ```

5. **Run the Application**:
   ```bash
   uvicorn app.main:app --reload
   ```

6. **Test the API**:
   Open `http://127.0.0.1:8000/docs` in your browser to test `/upload`, `/chat`, and `/stats` via the autogenerated Swagger UI.

## Technical Decisions

### Chunking Strategy Rationale

For text splitting, I utilized the **`RecursiveCharacterTextSplitter`** from LangChain with a `chunk_size` of 500 and a `chunk_overlap` of 100. 

**Why this approach?**
A purely naive chunking strategy (e.g., splitting every 500 characters blindly) often cuts paragraphs and sentences right in the middle, destroying the grammatical structure and semantic meaning of the text. This severely degrades embedding quality. 

The recursive splitter elegantly solves this by splitting along semantic boundaries first: it tries to split on paragraphs (`\n\n`), then lines (`\n`), then sentences, and only falls back to raw characters if an unbreakable block is too large. When combined with a 100-character overlap, consecutive chunks retain shared context, ensuring that search queries hovering cleanly across the boundaries of two thoughts don't result in isolated, out-of-context retrieval.

### Hallucination Handling
The system heavily grounds the LLM context. In `prompt.py`, the system prompt strictly dictates that the model must reply with exactly `"I couldn't find information about that in the uploaded content."` if the retrieved text chunks do not contain the answer. Furthermore, the `POST /chat` streaming endpoint natively detects this generic fallback phrase in real-time to flag and persist the query as an "Unanswered Question" in our SQLite metrics table.

### Multi-Bot Isolation
To ensure the multi-client vectors never bleed into one another, the ChromaDB implementation bypasses using a global collection. Every `/upload` POST generates a unique `bot_id` UUID, and dynamically spins up a completely sandboxed `/chroma_db/{bot_id}` directory containing an isolated collection.

## What I'd Do Differently (Future Scope)

With more time and a larger scope, I would improve the architecture by implementing **Hybrid Search Retrieval (Keyword + Semantic search)**.

Currently, standard vector dense retrieval relies purely on semantic closeness. This is great for broad concepts (e.g., "how does this app work?"), but often struggles with exact-keyword matching (e.g., searching for a very specific ID code, an acronym, or a rare noun). By introducing a hybrid retrieval pipeline that utilizes something like BM25 alongside dense vector embeddings, we would guarantee that hyper-specific keyword searches recall the exact PDF pages required.

**LangSmith Observability:**
I would also implement LangSmith observability for better latency monitoring, effective caching, and end-to-end tracing of LLM calls. This would provide deep insights into token usage, performance bottlenecks, and overall system reliability.

Additionally, for a production scale system, I would migrate the local file-based ChromaDB storage over to a fully managed vector index (like Pinecone or Qdrant) and implement a robust queueing system (like Celery/Redis) to handle heavy asynchronous PDF parsing for concurrent users.
